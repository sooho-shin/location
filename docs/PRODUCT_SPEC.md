# LOCATION 제품 기획서

## 문서 목적

이 문서는 LOCATION 프로젝트의 제품 기준점이다. 기능 추가, UI 변경, API 변경, 배포 설정 변경을 하기 전에 먼저 이 문서를 확인한다.

코드와 이 문서가 충돌하면 다음 순서로 처리한다.

1. 현재 사용자 요청이 기획 변경인지 확인한다.
2. 기획 변경이면 이 문서를 먼저 갱신한다.
3. 기획 변경이 아니면 이 문서의 방향을 우선한다.
4. 판단이 애매하면 구현을 진행하기 전에 사용자에게 확인한다.

## 제품 정의

LOCATION은 사용자의 현재 위치와 관심 카테고리를 기준으로 한국의 장소를 추천하고, Google Maps 위에 결과를 표시하는 지도 기반 장소 추천 웹 앱이다.

핵심은 일반 검색 결과를 나열하는 것이 아니라, 사용자의 위치와 맥락에 맞는 장소 후보를 AI가 정리해 지도에서 바로 탐색하게 만드는 것이다.

## 타겟 사용자

우선 타겟은 한국을 방문하거나 방문 예정인 일본인 사용자다. 다만 현재 구현은 한국어 UI를 기준으로 동작하므로, 다국어 전환은 단계적으로 확장한다.

우선순위:

1. 모바일 사용자가 현재 위치 주변 장소를 빠르게 찾는 경험
2. 일본인 관광객이 이해하기 쉬운 카테고리와 장소 설명
3. 한국 사용자도 어색하지 않은 장소 정보와 지도 UX

## 핵심 가치

- 현재 위치 기반 추천: 사용자가 직접 지역명을 입력하지 않아도 주변 장소를 탐색한다.
- 실제 장소 기반 추천: Google Places 후보가 있으면 실제 장소 후보 안에서 추천한다.
- AI 설명 보강: Gemini가 장소 후보를 카테고리 맥락에 맞게 정리한다.
- 지도 중심 탐색: 추천 결과를 카드 목록만이 아니라 지도 마커와 상세 패널로 확인한다.
- 모바일 우선: 첫 화면은 지도와 바텀시트 중심의 사용 화면이어야 한다.

## 주요 사용자 흐름

1. 사용자가 앱에 접속한다.
2. 브라우저 위치 권한을 요청한다.
3. 위치 권한이 허용되면 현재 위치를 기준으로 지도 중심을 설정한다.
4. 위치 권한이 거부되거나 실패하면 서울 시청 좌표를 기본값으로 사용한다.
5. 사용자가 바텀시트에서 카테고리를 선택한다.
6. 프론트엔드가 백엔드 `POST /api/recommend`에 카테고리, 키워드, 위도, 경도를 보낸다.
7. 백엔드는 Google Places 후보를 가져오고, Gemini로 추천 설명과 순서를 정리한다.
8. 프론트엔드는 추천 장소를 Google Maps 마커로 표시한다.
9. 사용자가 마커 또는 클러스터를 선택한다.
10. 장소 상세 패널에서 주소, 평점, 영업시간, 전화번호, 리뷰, 길찾기 링크를 확인한다.

## 현재 카테고리

현재 제품에 포함된 카테고리는 다음 두 개다.

| ID | 표시 이름 | 추천 키워드 |
| --- | --- | --- |
| `kpop` | 케이팝 헌터스 | K-pop 관련 명소, 아이돌 연습실, 엔터테인먼트 회사, 굿즈샵 |
| `ramen` | 한강라면 | 한강 공원 편의점, 라면 먹을 수 있는 곳, 한강 피크닉 |

새 카테고리를 추가할 때는 다음 항목을 함께 정의한다.

- 안정적인 영문 `id`
- 사용자에게 보이는 이름
- Google Places 검색에 쓸 키워드
- 카테고리 대표 이미지
- 지도 마커 색상
- 상세 패널에서 사용할 설명 문구

## 추천 정책

백엔드는 Google Places API와 Gemini API를 결합한다.

- `GOOGLE_PLACES_API_KEY`가 있으면 Places Text Search 후보를 먼저 가져온다.
- 후보가 있으면 Gemini는 후보 목록 안에서만 추천 결과를 골라야 한다.
- 후보가 없으면 Gemini가 실제 존재하는 장소를 추천할 수 있다.
- 추천 개수는 최대 10개다.
- Gemini 응답은 JSON 객체만 허용한다.
- 후보가 있는데 Gemini JSON 파싱이 실패하면 Google Places 후보를 그대로 반환한다.
- `GEMINI_API_KEY`가 없고 후보도 없으면 추천은 실패해야 한다.

금지 사항:

- 존재 여부가 불명확한 장소를 실제 장소처럼 강하게 표현하지 않는다.
- 사용자 위치와 무관한 먼 장소를 기본 추천으로 섞지 않는다.
- API 키, 원본 Gemini 응답, 개인 위치 정보를 로그에 과도하게 남기지 않는다.

## 지도 정책

지도 스택은 Google Maps JavaScript API를 기준으로 한다.

- 프론트엔드 지도 키는 `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY`를 사용한다.
- 백엔드 Places 검색 키는 `GOOGLE_PLACES_API_KEY`를 사용한다.
- 두 키는 분리해서 관리한다.
- 지도는 전체 화면 경험의 기반 레이어다.
- Header와 BottomSheet는 지도 위에 놓이는 조작 레이어다.
- 마커는 현재 위치, 단일 장소, 클러스터를 구분한다.
- 지도 라이브러리는 브라우저 전용이므로 Next.js SSR과 충돌하지 않도록 동적 로딩한다.

## UI/UX 원칙

- 첫 화면은 랜딩 페이지가 아니라 실제 지도 탐색 화면이어야 한다.
- 모바일 우선으로 설계한다.
- Header는 브랜드와 언어 선택만 담는 가벼운 상단 레이어다.
- BottomSheet는 카테고리 선택과 추천 결과 탐색의 중심이다.
- 지도 마커, 클러스터 패널, 상세 패널은 서로 시각적으로 겹치지 않게 한다.
- 카테고리 카드는 실제 맥락이 느껴지는 이미지를 사용한다.
- 최종 UI에 단색 placeholder 이미지를 남기지 않는다.
- 텍스트는 작은 화면에서 잘리거나 겹치지 않아야 한다.
- 사용자에게 보이는 문구는 기본적으로 한국어를 유지하되, 일본인 타겟 확장 시 일본어 UX를 별도 기획으로 반영한다.

## 언어 정책

현재 UI는 한국어를 기본으로 한다. Header에는 언어 선택 UI가 있지만 실제 번역 시스템은 아직 완성된 기능으로 보지 않는다.

다국어 기능을 구현할 때는 다음 순서를 따른다.

1. 지원 언어와 기본 언어를 기획 문서에 명시한다.
2. UI 문자열을 코드에서 분리한다.
3. 카테고리 이름, 추천 키워드, 에러 메시지, 상세 패널 문구를 함께 번역한다.
4. Gemini 프롬프트의 응답 언어 정책을 명확히 한다.

## 환경 변수

프론트엔드:

```env
NEXT_PUBLIC_API_URL=http://localhost:3001
NEXT_PUBLIC_GOOGLE_MAPS_API_KEY=your_google_maps_api_key_here
```

백엔드:

```env
GEMINI_API_KEY=your_gemini_api_key_here
GOOGLE_PLACES_API_KEY=your_google_places_api_key_here
PORT=3001
NODE_ENV=development
```

## 성공 기준

로컬 개발 기준의 성공 상태:

- 프론트엔드가 `http://localhost:3000`에서 열린다.
- 백엔드 `/health`가 정상 응답한다.
- Google Maps가 첫 화면에 표시된다.
- 카테고리 선택 시 추천 요청이 성공한다.
- 추천 결과가 지도 마커로 표시된다.
- 장소 상세 패널이 열리고, Places 상세 정보가 있으면 추가 정보가 표시된다.

## 향후 기획 메모

- 일본어 UI와 추천 설명을 제품 레벨 기능으로 정리한다.
- 카테고리 관리 구조를 하드코딩에서 설정 파일 또는 백엔드 관리 구조로 분리한다.
- Google Places 사진 API 또는 자체 이미지 정책을 정한다.
- 위치 권한 거부 상태 전용 화면을 만든다.
- 추천 실패, API quota 초과, 결제 비활성화 상태의 사용자 안내를 정리한다.
